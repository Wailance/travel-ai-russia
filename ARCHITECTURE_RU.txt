# Как проект работает внутри

Этот документ объясняет внутреннюю логику проекта простым языком.

---

## 1) Общая схема

Проект состоит из двух частей:

- **Frontend** (папка `public` + корневой `index.html`)  
  Показывает форму, отправляет запрос на API, рисует результат.
- **Backend** (`server.js`)  
  Получает параметры поездки, обращается к GigaChat, нормализует данные, считает логистику и бюджет, возвращает готовый JSON.

---

## 2) Поток запроса от пользователя

1. Пользователь заполняет форму:
   - дни,
   - город старта / окончания,
   - бюджет,
   - флаги (`Нужно проживание`, `Есть свое авто`).
2. `public/app.js` отправляет `POST /api/plan`.
3. `server.js`:
   - проверяет входные данные,
   - оценивает минимальный бюджет,
   - если бюджет слишком мал — сразу возвращает спец-ответ без вызова модели,
   - иначе строит prompt и вызывает GigaChat.
4. Ответ модели приводится к стабильному формату:
   - `normalizePlan(...)`,
   - `buildLogistics(...)`,
   - `rebalanceBudgetByPreferences(...)`,
   - `enrichPlanWithDayMaps(...)`.
5. Frontend рендерит:
   - путь,
   - расписание по дням,
   - логистику по сегментам,
   - бюджет,
   - графики.

---

## 3) Главные модули в backend (`server.js`)

## `buildPrompt(...)`
- Формирует текстовый запрос к GigaChat.
- Передает ограничения пользователя (дни, города, бюджет, проживание, свое авто).
- Просит строгий JSON-формат.

## `requestGigaChat(...)`
- Делает запрос к GigaChat API.
- Поддерживает OAuth/токен через env.
- Есть retry на `429`.

## `parseModelJson(...)` и `repairJsonWithModel(...)`
- Убирают markdown/мусор.
- Пытаются распарсить JSON.
- Если JSON битый — запускается попытка “починки” через модель.

## `normalizePlan(...)`
- Гарантирует нужное число дней.
- Добавляет недостающие категории (транспорт/еда/активности/проживание).
- Выставляет ориентировочные стоимости там, где нет явных цен.

## `buildLogistics(...)`
- Геокодирует города.
- Строит сегменты между городами.
- Выбирает транспорт (`car/train/bus/plane`) по правилам и дистанции.
- Считает расстояние/время/стоимость.
- Для `plane` может пробовать live-тариф (если есть токен источника).

## `rebalanceBudgetByPreferences(...)`
- Собирает итоговый бюджетный план:
  - transport / hotel / food / activities / reserve / total.
- Учитывает флаги:
  - `needAccommodation`,
  - `hasOwnCar`.
- Выставляет `budgetReality` (`low|ok|high`) и сообщение.

## `estimateMinimumBudget(...)`
- Быстрый sanity-check перед вызовом модели.
- Если бюджет слишком мал для базового маршрута — API честно останавливает генерацию.

## `createFallbackPlan(...)`
- Резервный план, если внешняя модель недоступна.
- Нужен, чтобы пользователь не видел “пустой” результат при сбоях провайдера.

---

## 4) Frontend логика (`public/app.js`)

Ключевые функции:

- `getApiBaseUrl()` — берет URL backend из `public/config.js`.
- `renderPlan(...)` — главный рендер данных плана.
- `renderLowBudget(...)` — отдельный режим для “бюджет слишком мал”.
- `renderLogistics(...)` — таблица сегментов с типом транспорта и источником цены.
- `renderCharts(...)` — простые графики по бюджету и расходам по дням.

---

## 5) Карты

- Основная карта маршрута: `Yandex Static Maps`.
- Координаты городов/точек:
  - геокодинг через Nominatim,
  - fallback через `DEFAULT_CITY_COORDS`.

---

## 6) Безопасность и ограничения

В backend уже включено:

- `helmet` (базовые secure headers),
- CORS-ограничение по `FRONTEND_ORIGIN`,
- rate-limit на `/api/*`,
- `.env` не коммитится.

---

## 7) Переменные окружения (главные)

- `GIGACHAT_TOKEN` или `GIGACHAT_AUTH_KEY`
- `GIGACHAT_USE_OAUTH`
- `GIGACHAT_INSECURE_TLS` (только при необходимости)
- `FRONTEND_ORIGIN`
- `GIGACHAT_MODEL`
- (опционально для live авиатарфа) `TRAVELPAYOUTS_TOKEN`

---

## 8) Файлы и ответственность

- `server.js` — вся серверная бизнес-логика.
- `public/index.html` — структура UI.
- `public/styles.css` — визуал.
- `public/app.js` — клиентская логика и рендер.
- `public/config.js` — URL backend для фронта.
- `render.yaml` — конфигурация деплоя на Render.

---

## 9) Что важно помнить при доработках

- Любые новые поля формы нужно протягивать по всей цепочке:
  `index.html -> app.js payload -> server.js req.body -> расчеты -> response -> UI`.
- Логику цен лучше менять централизованно:
  - `estimateTransportCost(...)`,
  - `rebalanceBudgetByPreferences(...)`,
  - `estimateMinimumBudget(...)`.
- Если меняется формат ответа API, обязательно синхронизировать `renderPlan(...)` на фронте.
